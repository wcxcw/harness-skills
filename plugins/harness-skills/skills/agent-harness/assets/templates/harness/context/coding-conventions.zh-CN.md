# 代码约定

记录 Agent 必须遵守的项目专属约定。

## 风格

- [命名、格式化、组件或模块约定]
- 优先沿用项目既有结构和本地 helper API，避免随意引入新模式。
- 根应用文件、路由壳、controller 和 service 应保持编排职责。
- 当 UI、状态、数据访问、业务规则和格式化逻辑开始挤在一个大文件里时，应拆分代码。

## 前端结构

- `App.vue`、React 顶层 app 文件等根组件应作为组合壳；除非项目明确采用单文件 demo 模式，不要承载完整业务。
- 框架支持时，把可复用状态逻辑移动到 composable、hook 或 store。
- 把服务端请求放到既有 API/data helper 中，不要直接塞进视图组件。
- 本地可运行时，用户可见的浏览器变化应使用真实浏览器验证。

## 后端结构

- 项目存在 service 或数据层时，handler/controller 应保持轻量。
- 在信任边界验证输入，并保留授权、schema 和兼容性假设。
- 项目存在日志设施时，重要外部调用、状态变化、后台任务和异常路径应记录日志。

## 错误处理

- [项目专属要求]
- 不要吞掉错误，或把有用诊断替换成模糊的成功/失败状态。
- 跳过错误路径验证时，应记录为残留风险。

## 测试

- [测试所在位置和测试风格]

## 注释

- 只为非显而易见的业务规则、边界条件、不变量、权衡和集成约束添加注释。
- 不要添加只复述语法的注释。

## Review 要求

- 修改范围保持在当前 spec 内。
- 优先使用既有模式，避免新抽象。
- 只有在能解释非显而易见逻辑时才添加注释。
- 代码质量问题不自动归类为“风格”；可维护性、日志、错误处理和可测试性缺口影响后续工作时，应作为 review 发现处理。
