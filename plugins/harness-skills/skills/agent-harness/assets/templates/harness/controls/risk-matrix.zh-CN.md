# 风险矩阵

使用本矩阵选择最小安全 tier、review 深度和验证范围。

## 风险因素

| 风险 | 示例 | 最低处理 |
| --- | --- | --- |
| 数据丢失 | 删除/更新流程、迁移、持久化修改 | `full` run、后端质量 review、明确 rollback 记录 |
| 安全/权限 | 登录、授权、密钥、权限 | `full` run、安全重点 review、测试或手动证据 |
| API/数据契约 | 请求/响应结构、schema、外部集成 | `standard` 或 `full`，契约验证 |
| 架构 | 模块边界、共享状态、依赖方向 | 引入新方向时使用 `full` |
| 前端 UX | 导航、核心流程、可访问性、响应式布局 | 前端质量 review 和浏览器证据 |
| 可观测性 | 后端副作用、后台任务、外部调用 | 后端质量 review 和日志检查 |
| 依赖/CI | 包变更、工具链、构建/测试流水线 | `standard` 或 `full`，构建和 rollback 证据 |
| 上下文规模 | 大型 context 文件、长期历史 run、项目事实不清 | 实现前使用 context-budget |

## Tier 规则

- 只有完全满足 `gates.md` 的微小改动标准时，才使用 no-run。
- `xs` 用于低风险但仍需要证据的小改动。
- `standard` 用于普通功能、范围明确的 bugfix 和多文件修改。
- 数据、安全、架构、产品方向、CI 或宽泛 UX 行为存在高风险时，使用 `full`。

## Review 规则

- 应用代码变更需要代码质量 review；跳过时必须记录原因。
- 前端变化涉及 UI、状态、路由、可访问性或浏览器行为时，使用前端质量 review。
- 后端/API/service 变化涉及数据、错误、日志、副作用或外部调用时，使用后端质量 review。
- Critical 和 Important review 发现会阻塞完成，直到修复或被用户接受为残留风险。

## 证据规则

- 风险越高，证据越强：lint/build/test 加定向手动、浏览器或 API 验证。
- 跳过检查必须说明无法运行的原因，以及使用了什么替代证据。
- 无法判断风险时，停止并询问，或把未知项记录为阻塞决策。
