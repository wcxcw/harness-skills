# 边界

## 始终

- 基于当前 spec 和 plan 工作。
- 编辑前先阅读相关文件。
- 保持修改范围贴合当前需求。
- 记录命令和验证结果。
- 优先选择满足 spec 的最简单改动。
- 保留有价值的既有模式，不随意创造新模式。

## 先询问

- 范围超出当前 spec。
- 需求与现有架构冲突。
- 验证需要外部凭据或生产系统。
- 改动会引入新依赖、新框架、新服务或宽泛抽象。

## 禁止

- 编造未经确认的命令。
- 隐藏失败测试或跳过的检查。
- 修改生成文件、供应商文件或锁文件，除非必要且已理解影响。
- 为未请求的未来需求添加猜测式抽象。
- 只因为“代码已修改”就认为任务完成，必须有验证证据。
